# TRM-Ag Tuning Harness Requirements

## 1. Purpose

This document defines requirements for an iterative tuning harness that adjusts
runtime parameters for `TRM-Ag` evaluation loops while avoiding known
anti-patterns (threshold hacking, overfitting to a single seed, and unstable
parameter oscillation).

The harness must operate on top of existing
`trm_pipeline.experiment_harness` contracts.

## 2. Scope

In scope:

- repeated `run -> evaluate -> revise` execution for `TRM-Ag` candidate mode
- bounded runtime-parameter updates
- family-track-aware revisions for blocked tracks
- deterministic artifact logging for auditability

Out of scope:

- changing model weights
- changing acceptance threshold definitions
- replacing existing `run_contract` evaluation semantics

## 3. Functional Requirements

### R-001 Contract-first execution

The tuner MUST accept an existing experiment contract and derive each tuning
round from that contract.

### R-002 Baseline invariance

The tuner MUST NOT rewrite baseline mode, acceptance criteria, or promotion
targets during tuning rounds.

### R-003 Bounded updates

Each round MUST apply only bounded parameter deltas, with explicit min/max
clamps per tunable parameter.

### R-004 Track-local revisions

When family tracks are defined, parameter revisions SHOULD be applied only to
blocked required tracks, not all tracks.

### R-005 Monotonic evidence rule

The tuner MUST track a primary score and stop if no meaningful improvement is
observed for configured patience.

### R-006 Promotion stop rule

The tuner MUST stop immediately when required tracks pass promotion gates.

### R-007 Audit artifacts

The tuner MUST write per-round and final summaries under a dedicated tuning
artifact root.

Minimum artifacts:

- `autotune/tune_summary.json`
- `autotune/round_XX/contract.json`
- per-round run/eval artifacts generated by `run_contract`

### R-008 Safe failure mode

If revision proposals become empty (no safe update), the tuner MUST stop with
`status=stalled` rather than inventing unconstrained edits.

## 4. Non-functional Requirements

### N-001 Reproducibility

Round outputs MUST preserve the contract seed range semantics and remain
re-runnable from artifacts.

### N-002 Testability

Core tuning logic MUST be unit tested without requiring heavy runtime sweeps.

### N-003 Backward compatibility

Existing `plan`, `run`, and `evaluate` commands MUST continue to work unchanged.

## 5. Acceptance Criteria

The requirement is accepted when:

1. tuning command exists and writes `autotune/tune_summary.json`
2. at least one unit test verifies bounded parameter clamping
3. at least one unit test verifies stop-on-promotion behavior
4. existing experiment harness tests remain green
